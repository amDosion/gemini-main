基于当前的代码库，所有的模式（聊天、生图、视频、PDF提取等）以及特殊功能（思考过程、联网搜索）的数据保存都统一遵循同一个数据模型，并通过 useSessions 和 HybridDB 进行持久化。
核心在于 Message 接口的设计（位于 types.ts），它是一个极其灵活的容器，能够容纳各种类型的数据。
以下是详细的保存机制分析：
1. 统一的数据容器：Message 对象
无论什么模式，每一条记录本质上都是一个 Message 对象。这个对象被存储在 ChatSession 的 messages 数组中。

// types.ts (核心定义)
export interface Message {
  id: string;
  role: Role;          // 'user' | 'model'
  content: string;     // 核心文本内容 (包括思考过程、PDF结果JSON)
  attachments?: Attachment[]; // 多媒体附件 (图片、视频、音频文件)
  groundingMetadata?: GroundingMetadata; // 联网搜索数据
  // ... 其他字段
}

2. 不同模式的具体保存方式
A. 思考过程 (Thinking Process)
Gemini 2.0 Flash Thinking 等模型返回的思考过程并不是存储在独立的字段中，而是作为原始文本的一部分存储在 content 字段里。
保存格式：字符串包含 XML 标签。
message.content = "<thinking>我需要先分析用户的意图...</thinking> 根据分析，答案是..."
如何展示：useMessageProcessor.ts 在渲染时实时解析字符串，提取 <thinking> 标签内的内容传给 ThinkingBlock 组件。数据库里存的就是带标签的原始字符串。

B. 网络搜索 (Google Search Grounding)
当启用 Search 工具时，返回的搜索源数据存储在专用的 groundingMetadata 字段中。
保存位置：message.groundingMetadata
数据结构：
{
  "webSearchQueries": ["Gemini 2.5 release date"],
  "groundingChunks": [
    { "web": { "uri": "https://google.com/...", "title": "Google Blog" } }
  ]
}

持久化：这个对象会被完整序列化为 JSON 存入 LocalStorage 或数据库。

C. 生成/编辑/扩展模式 (Gen / Edit / Expand / Video / Audio)
这些模式生成的多媒体内容不存放在 content 中，而是存放在 attachments 数组中。
保存位置：message.attachments
保存格式：
Base64 Data URI：目前大多数生成结果（Google Imagen, Veo, TTS）通过 SDK 返回的是二进制数据。前端将其转换为 Base64 字符串（data:image/png;base64,...）存入 url 字段。
优势：数据完全包含在 JSON 中，离线可用。
劣势：LocalStorage 有 5MB 限制，存几张大图就会爆。（注：当前代码逻辑是存 Base64，如果是生产环境通常会上传到 S3 存 HTTP URL）。
Content 字段：此时 content 字段通常只存储简单的描述，如 "Generated video: 'A cat driving'"。
D. PDF 提取模式 (PDF Extract)
这是一个特殊情况。虽然是结构化数据，但为了通用性，它被序列化为字符串存入 content。
保存位置：message.content
保存格式：JSON 字符串。

message.content = JSON.stringify({
  success: true,
  data: { invoice_number: "123", total: 100 }
});

渲染时：PdfExtractView 会尝试 JSON.parse(message.content)，如果成功则渲染表格，否则渲染普通文本。

    4. 总结表格
功能/模式	核心数据	存储位置 (Message 属性)	数据格式	备注
思考过程	<thinking>...	content	String (XML)	渲染时通过正则提取，不存为独立字段

视频生成	视频数据	attachments	Base64 URI	mimeType: "video/mp4"
语音生成	音频数据	attachments	Base64 URI	mimeType: "audio/wav"

这种设计的优点是前端组件的高度复用。MessageItem 组件只需要判断 content 是否包含 JSON、attachments 是否有图片、groundingMetadata 是否存在，就可以自动渲染出极其复杂的 UI，而不需要为每种模式设计独立的数据库表结构。
