# 模式切换联动问题分析报告

## 问题描述

从A模式切换到B模式后，如果B模式中没有符合C模式的模型，会导致无法切换到C模式。

## 问题根源分析

### 1. 数据流追踪

#### 1.1 模型列表的生成流程

```
useModels Hook (frontend/hooks/useModels.ts)
├── availableModels: 完整的模型列表（从API获取，未过滤）
├── filteredModelsByMode: 根据当前 appMode 过滤后的模型列表
│   └── filterModelsByAppMode(availableModels, appMode)
└── visibleModels: 排除隐藏模型后的可见模型列表
    └── filteredModelsByMode.filter(m => !hiddenModelIds.includes(m.id))
```

**关键代码位置：**
- `frontend/hooks/useModels.ts:107-109` - `filteredModelsByMode` 基于当前 `appMode` 过滤
- `frontend/hooks/useModels.ts:328-330` - `visibleModels` 基于 `filteredModelsByMode` 生成

#### 1.2 ModeSelector 的可用性判断

```
ModeSelector Component (frontend/components/chat/input/ModeSelector.tsx)
├── 接收 visibleModels 作为 prop
├── modeAvailability 计算逻辑（第15-81行）
│   └── 基于 visibleModels 判断每个模式的可用性
└── 如果 visibleModels 中没有符合某个模式的模型，该模式会被禁用
```

**关键代码位置：**
- `frontend/components/chat/input/ModeSelector.tsx:15-81` - `modeAvailability` 基于 `visibleModels` 计算
- `frontend/components/chat/input/ModeSelector.tsx:16` - 使用 `visibleModels.length > 0 ? visibleModels : [currentModel]`

### 2. 问题场景重现

#### 场景示例

1. **初始状态：**
   - 当前模式：`chat`
   - `availableModels`: [gemini-2.5-pro, imagen-4.0, veo-3.1, ...] (完整列表)
   - `visibleModels`: [gemini-2.5-pro, ...] (chat模式过滤后的列表)
   - `modeAvailability`: 所有模式都可用（因为完整模型列表包含各种类型的模型）

2. **切换到 image-gen 模式：**
   - `appMode` 变为 `image-gen`
   - `filteredModelsByMode`: [imagen-4.0, ...] (只包含图像生成模型)
   - `visibleModels`: [imagen-4.0, ...] (只包含图像生成模型)
   - `modeAvailability` 重新计算：
     - `hasVideoModels`: false (visibleModels 中没有 veo 模型)
     - `hasDeepResearchModels`: false (visibleModels 中没有 deep-research 模型)
     - `video-gen`: disabled ❌
     - `deep-research`: disabled ❌

3. **尝试切换到 video-gen 模式：**
   - 由于 `modeAvailability.video-gen = false`，按钮被禁用
   - 用户无法切换到 video-gen 模式 ❌

### 3. 核心问题

**`ModeSelector` 使用错误的模型列表来判断模式可用性：**

- **当前实现：** 使用 `visibleModels`（已按当前模式过滤）
- **正确实现：** 应该使用 `availableModels`（完整模型列表）

**原因：**
- `visibleModels` 是当前模式下的可见模型，不包含其他模式的模型
- 模式可用性应该基于完整的模型列表判断，而不是当前模式下的模型列表

## 影响范围

### 受影响的模式切换场景

1. **从专用模式切换到其他专用模式：**
   - `image-gen` → `video-gen` ❌
   - `video-gen` → `deep-research` ❌
   - `audio-gen` → `image-gen` ❌

2. **从通用模式切换到专用模式：**
   - `chat` → `video-gen` ✅ (因为 chat 模式包含所有模型)
   - `chat` → `image-gen` ✅

3. **从专用模式切换到通用模式：**
   - `image-gen` → `chat` ✅ (chat 始终可用)
   - `video-gen` → `chat` ✅

### 代码影响范围

- **主要影响组件：**
  - `frontend/components/chat/input/ModeSelector.tsx` - 模式选择器
  - `frontend/hooks/useModels.ts` - 模型管理 Hook

- **相关组件：**
  - `frontend/components/chat/InputArea.tsx` - 输入区域（传递 visibleModels）
  - `frontend/components/views/ChatView.tsx` - 聊天视图（传递 visibleModels）
  - `frontend/App.tsx` - 主应用（使用 useModels）

## 解决方案

### 方案1：修改 ModeSelector 使用 availableModels（推荐）

**修改位置：** `frontend/components/chat/input/ModeSelector.tsx`

**修改内容：**
1. 添加 `availableModels` prop（完整模型列表）
2. 修改 `modeAvailability` 计算逻辑，使用 `availableModels` 而不是 `visibleModels`

**优点：**
- 模式可用性判断准确
- 不影响现有功能
- 最小化代码修改

**缺点：**
- 需要传递额外的 prop

### 方案2：在 useModels 中提供未过滤的模型列表

**修改位置：** `frontend/hooks/useModels.ts`

**修改内容：**
1. 添加 `allVisibleModels`（排除隐藏模型但不过滤模式）
2. 返回给调用者使用

**优点：**
- 集中管理模型列表
- 减少 prop 传递

**缺点：**
- 需要修改多个调用点

### 方案3：在 ModeSelector 内部重新计算可用性

**修改位置：** `frontend/components/chat/input/ModeSelector.tsx`

**修改内容：**
1. 接收 `availableModels` 和 `hiddenModelIds`
2. 内部计算每个模式的可用性（排除隐藏模型但不过滤模式）

**优点：**
- 封装性好
- 逻辑集中

**缺点：**
- 需要传递更多 props

## 推荐实现方案

**推荐使用方案1：修改 ModeSelector 使用 availableModels**

### 实现步骤

1. **修改 ModeSelector 接口：**
```typescript
interface ModeSelectorProps {
  mode: AppMode;
  setMode: (mode: AppMode) => void;
  currentModel?: ModelConfig;
  availableModels?: ModelConfig[];  // 新增：完整模型列表
  hiddenModelIds?: string[];        // 新增：隐藏模型ID列表
  visibleModels?: ModelConfig[];    // 保留：用于向后兼容（可选）
}
```

2. **修改 modeAvailability 计算：**
```typescript
const modeAvailability = useMemo(() => {
  // 使用 availableModels 而不是 visibleModels
  const models = (availableModels || visibleModels || [])
    .filter(m => !hiddenModelIds?.includes(m.id));
  
  // ... 其余逻辑保持不变
}, [availableModels, visibleModels, hiddenModelIds, currentModel]);
```

3. **更新调用点：**
   - `frontend/components/chat/InputArea.tsx` - 传递 `availableModels` 和 `hiddenModelIds`
   - `frontend/components/views/ChatView.tsx` - 传递 `availableModels` 和 `hiddenModelIds`
   - 其他使用 ModeSelector 的视图组件

## 总结

**问题本质：** `ModeSelector` 使用已按当前模式过滤的 `visibleModels` 来判断其他模式的可用性，导致从专用模式无法切换到其他专用模式。

**解决方案：** 修改 `ModeSelector` 使用完整的 `availableModels` 来判断模式可用性，确保所有模式都能正确显示可用状态。
