# GEN模式跳转Edit/Expand模式附件显示需求文档

## 一、问题背景

### 1.1 当前场景

用户在 **GEN 模式**生成图片后，立即点击 **Edit/Expand 按钮**跳转到 Edit/Expand 模式时，附件的显示可能存在以下问题：

1. **异步查询延迟**：如果图片刚生成，`uploadStatus === 'pending'`，会触发异步查询后端获取云 URL，导致显示延迟
2. **Base64 URL 被不必要查询**：Base64 URL 已经完整可用，但仍会触发 `tryFetchCloudUrl` 查询后端
3. **HTTP 临时 URL 过期风险**：如果上传未完成，可能使用已过期的 HTTP 临时 URL，导致图片无法显示
4. **查找失败场景**：如果 `findAttachmentByUrl` 找不到历史附件，且传入的 URL 不可用，无法显示

### 1.2 用户期望

- ✅ **即时显示**：点击 Edit/Expand 按钮后，图片应该立即显示在目标模式的画布上
- ✅ **无需等待**：不应该因为查询后端而延迟显示
- ✅ **稳定可靠**：无论上传状态如何，都应该能够显示图片

## 二、功能需求

### 2.1 核心需求

**需求 R1：优化附件 URL 处理逻辑**

- **描述**：
  - 在 `useImageHandlers` 中，当用户点击 Edit/Expand 按钮时，应该**立即使用传入的 URL**（Base64 或 HTTP URL）显示图片
  - HTTP URL（包括 AI 提供商返回的临时 URL）可以立即显示，不需要等待查询后端
  - 查询后端应该在后台异步进行，不阻塞初始显示
  - **查询目的**：获取永久云存储 URL（如果上传已完成），用于后续 API 调用
- **优先级**：高
- **设计意图分类**：
  - 🚀 **加速显示**：立即使用传入的 URL，不等待查询后端
  - 🏗️ **有意设计**：查询目的明确（获取永久云存储 URL，而不是验证 URL 是否可用）
  - 🔄 **避免多次查询**：HTTP URL（包括临时 URL）可以立即显示，不需要查询验证
- **验收标准**：
  - Base64 URL 应该直接使用，立即显示，不查询后端
  - HTTP URL（包括临时 URL）应该直接使用，立即显示，不等待查询
  - 查询在后台异步进行，不阻塞 `setInitialAttachments`
  - 查询目的：获取永久云存储 URL（如果上传已完成），用于后续 API 调用

**需求 R2：避免不必要的后端查询和阻塞**

- **描述**：
  - `tryFetchCloudUrl` 不应该对 Base64 URL 和 Blob URL 进行查询
  - HTTP URL（包括 AI 提供商返回的临时 URL）应该直接使用，立即显示，不需要等待查询
  - 查询后端应该在后台异步进行，不阻塞初始显示
  - **查询目的**：获取永久云存储 URL（如果上传已完成），用于后续 API 调用，而不是为了验证 URL 是否可用
- **优先级**：高
- **设计意图分类**：
  - 🚀 **加速显示**：Base64 URL 和 Blob URL 直接使用，不查询后端
  - 🔄 **避免多次查询**：HTTP URL（包括临时 URL）直接使用，不需要查询验证
  - 🏗️ **有意设计**：只有 HTTP URL 且 `uploadStatus === 'pending'` 时才查询，避免不必要的请求
- **验收标准**：
  - Base64 URL 不触发 `tryFetchCloudUrl`
  - Blob URL 不触发 `tryFetchCloudUrl`
  - HTTP URL（包括临时 URL）直接使用，立即显示，不等待查询
  - 查询在后台异步进行，不阻塞 `setInitialAttachments`
  - `uploadStatus === undefined` 时不触发查询

**需求 R3：处理 HTTP 临时 URL 过期问题**

- **描述**：如果传入的是 HTTP 临时 URL，且上传未完成，应该能够降级使用 Base64 URL（如果可用）或等待上传完成
- **优先级**：中
- **降级策略**：优先使用 HTTP URL，如果加载失败（通过 `<img>` 的 `onError` 事件检测），自动切换到 Base64 URL
- **验收标准**：
  - 如果 HTTP 临时 URL 可用，直接使用
  - 如果 HTTP 临时 URL 不可用（加载失败），自动切换到 `tempUrl` 中的 Base64 URL
  - 如果都不可用，显示友好错误提示

**需求 R4：优化查找逻辑**

- **描述**：`findAttachmentByUrl` 应该能够可靠地找到刚生成的图片附件
- **优先级**：中
- **设计意图分类**：
  - 🏗️ **有意设计**：两级匹配策略，确保在各种场景下都能找到附件
  - 🔄 **避免多次查询**：通过精确匹配和兜底策略，避免重复创建附件
- **匹配策略**：
  1. **策略 1：精确匹配**：匹配 `att.url === targetUrl || att.tempUrl === targetUrl`
  2. **策略 2：Blob URL 兜底**：如果是 Blob URL 且未精确匹配，查找最近的有效云端图片附件（`uploadStatus === 'completed'` 且 `isHttpUrl(att.url)`）
- **验收标准**：
  - 即使 `messages` 中的 URL 被 `cleanAttachmentsForDb` 清空，也应该能够通过 `tempUrl` 精确匹配找到
  - Blob URL 场景下，如果精确匹配失败，能够通过兜底策略找到最近的有效云端图片附件

---

### 2.2 架构设计需求

**需求 R5：重载网页后的 URL 处理（有意设计）**

- **描述**：
  - 重载网页后，临时 URL（Base64、Blob URL）会失效
  - 重载后，只有永久云存储 URL（`uploadStatus === 'completed'`）会被保留
  - 前端自动使用永久云存储 URL 显示
- **优先级**：高（架构设计）
- **设计意图分类**：
  - 🏗️ **有意设计**：确保重载后能够正常显示图片，使用永久云存储 URL
  - 🔄 **避免多次查询**：重载后直接使用永久云存储 URL，不需要查询后端
- **处理流程**：
  1. 保存到数据库时：`cleanAttachmentsForDb` 清空 Base64 和 Blob URL，保留永久云存储 URL
  2. 重载后：从数据库加载，只有永久云存储 URL 被保留
  3. 前端显示：使用永久云存储 URL
- **验收标准**：
  - 重载后，Base64 URL 和 Blob URL 被清空
  - 重载后，只有永久云存储 URL（`uploadStatus === 'completed'`）被保留
  - 前端自动使用永久云存储 URL 显示

**需求 R6：后端上传后前端不更新会话（有意设计，避免多次查询）**

- **描述**：
  - 后端上传完成后，只更新数据库，不更新前端会话
  - 前端保持原始 URL（Base64 或 HTTP 临时 URL）用于显示
  - 避免前端重新渲染，提升用户体验
  - 重载后自动使用永久云存储 URL
- **优先级**：高（架构设计）
- **设计意图分类**：
  - 🏗️ **有意设计**：后端上传完成后，只更新数据库，不更新前端会话
  - 🔄 **避免多次查询**：保持原始 URL，避免查询后端获取云存储 URL
  - 🚀 **加速显示**：不更新前端会话，避免重新渲染，提升性能
- **处理流程**：
  1. AI 生成图片，返回 Base64 或 HTTP 临时 URL
  2. 前端立即显示（使用原始 URL）
  3. 后端异步上传到云存储
  4. 上传完成后，只更新数据库（url 字段更新为云存储 URL）
  5. 前端会话不更新，保持原始 URL
  6. 用户点击 Edit/Expand 按钮时，使用原始 URL 显示
  7. 重载网页后，使用永久云存储 URL 显示
- **验收标准**：
  - 后端上传完成后，只更新数据库，不更新前端会话
  - 前端保持原始 URL（Base64 或 HTTP 临时 URL）用于显示
  - 不触发前端重新渲染
  - 重载后自动使用永久云存储 URL

## 三、非功能需求

### 3.1 性能需求

- **响应时间**：点击 Edit/Expand 按钮后，图片应该在 **100ms 内**显示在画布上
- **网络请求**：减少不必要的后端查询，避免网络延迟

### 3.2 可靠性需求

- **可用性**：无论上传状态如何，都应该能够显示图片（使用 Base64 或 HTTP URL）
- **容错性**：如果 URL 不可用，应该显示友好的错误提示

### 3.3 兼容性需求

- **向后兼容**：修改不应该影响现有的其他功能
- **URL 类型支持**：支持 Base64、HTTP URL、临时代理 URL（如果存在）

### 7.3 关键设计决策

#### 决策 1：立即显示，不等待查询（🚀 加速显示）

**需求**：R1, R2

**原因**：提升用户体验，减少延迟

**实现**：
- Base64 URL 和 HTTP URL 都立即使用，不等待查询
- 查询在后台异步进行，不阻塞初始显示

#### 决策 2：后端上传后不更新前端会话（🏗️ 有意设计，🔄 避免多次查询）

**需求**：R6

**原因**：
- 避免前端重新渲染，提升性能
- 保持原始 URL，避免查询后端获取云存储 URL
- 重载后自动使用永久云存储 URL

**实现**：
- 后端上传完成后，只更新数据库，不更新前端会话
- 前端保持原始 URL（Base64 或 HTTP 临时 URL）用于显示
- 重载后，`cleanAttachmentsForDb` 会清空临时 URL，只保留永久云存储 URL

#### 决策 3：重载后使用永久云存储 URL（🏗️ 有意设计，🔄 避免多次查询）

**需求**：R5

**原因**：
- 临时 URL（Base64、Blob URL）在重载后会失效
- 永久云存储 URL 可以持久化，重载后仍然可用

**实现**：
- `cleanAttachmentsForDb` 清空 Base64 和 Blob URL，保留永久云存储 URL
- 重载后，从数据库加载，只有永久云存储 URL 被保留
- 前端自动使用永久云存储 URL 显示
