# GEN模式跳转Edit/Expand模式附件显示需求文档

## 一、问题背景

### 1.1 当前场景

用户在 **GEN 模式**生成图片后，立即点击 **Edit/Expand 按钮**跳转到 Edit/Expand 模式时，附件的显示可能存在以下问题：

1. **异步查询延迟**：如果图片刚生成，`uploadStatus === 'pending'`，会触发异步查询后端获取云 URL，导致显示延迟
2. **Base64 URL 被不必要查询**：Base64 URL 已经完整可用，但仍会触发 `tryFetchCloudUrl` 查询后端
3. **HTTP 临时 URL 过期风险**：如果上传未完成，可能使用已过期的 HTTP 临时 URL，导致图片无法显示
4. **查找失败场景**：如果 `findAttachmentByUrl` 找不到历史附件，且传入的 URL 不可用，无法显示

### 1.2 用户期望

- ✅ **即时显示**：点击 Edit/Expand 按钮后，图片应该立即显示在目标模式的画布上
- ✅ **无需等待**：不应该因为查询后端而延迟显示
- ✅ **稳定可靠**：无论上传状态如何，都应该能够显示图片

---

## 二、功能需求

### 2.1 核心需求

**需求 R1：优化附件 URL 处理逻辑**

- **描述**：在 `useImageHandlers` 中，当用户点击 Edit/Expand 按钮时，应该优先使用传入的 URL（Base64 或 HTTP URL），而不是立即查询后端
- **优先级**：高
- **验收标准**：
  - Base64 URL 应该直接使用，不查询后端
  - HTTP URL 应该直接使用，不查询后端（除非明确需要云 URL）
  - 只有在必要时（如上传已完成且有云 URL）才查询后端

**需求 R2：避免不必要的后端查询**

- **描述**：`tryFetchCloudUrl` 不应该对 Base64 URL 进行查询，因为 Base64 URL 已经完整可用
- **优先级**：高
- **验收标准**：
  - Base64 URL 不触发 `tryFetchCloudUrl`
  - 只有 HTTP URL 且 `uploadStatus === 'pending'` 时才查询后端

**需求 R3：处理 HTTP 临时 URL 过期问题**

- **描述**：如果传入的是 HTTP 临时 URL，且上传未完成，应该能够降级使用 Base64 URL（如果可用）或等待上传完成
- **优先级**：中
- **验收标准**：
  - 如果 HTTP 临时 URL 不可用，应该尝试使用 `tempUrl` 中的 Base64 URL
  - 如果都没有，应该显示错误提示

**需求 R4：优化查找逻辑**

- **描述**：`findAttachmentByUrl` 应该能够可靠地找到刚生成的图片附件
- **优先级**：中
- **验收标准**：
  - 即使 `messages` 中的 URL 被 `cleanAttachmentsForDb` 清空，也应该能够通过 `tempUrl` 或 `attachment.id` 找到

---

## 三、非功能需求

### 3.1 性能需求

- **响应时间**：点击 Edit/Expand 按钮后，图片应该在 **100ms 内**显示在画布上
- **网络请求**：减少不必要的后端查询，避免网络延迟

### 3.2 可靠性需求

- **可用性**：无论上传状态如何，都应该能够显示图片（使用 Base64 或 HTTP URL）
- **容错性**：如果 URL 不可用，应该显示友好的错误提示

### 3.3 兼容性需求

- **向后兼容**：修改不应该影响现有的其他功能
- **URL 类型支持**：支持 Base64、HTTP URL、临时代理 URL（如果存在）

### 用户故事 1：即时显示 Base64 图片

**作为** 用户  
**我希望** 在 GEN 模式生成图片后，立即点击 Edit 按钮时，图片能够立即显示在 Edit 模式的画布上  
**以便** 我可以立即开始编辑，无需等待

**验收标准**：
- 点击 Edit 按钮后，图片在 100ms 内显示
- 不触发不必要的后端查询
- 图片清晰可见

### 用户故事 2：处理 HTTP 临时 URL

**作为** 用户  
**我希望** 即使图片还在上传中，我也能够点击 Edit 按钮并看到图片  
**以便** 我可以立即开始编辑，无需等待上传完成

**验收标准**：
- 如果 HTTP 临时 URL 可用，直接使用
- 如果 HTTP 临时 URL 不可用，尝试使用 Base64 URL
- 如果都不可用，显示错误提示

### 用户故事 3：避免查询延迟

**作为** 用户  
**我希望** 点击 Edit/Expand 按钮时，不会因为查询后端而延迟显示  
**以便** 我可以立即看到图片并开始编辑

**验收标准**：
- Base64 URL 不触发后端查询
- HTTP URL 不触发不必要的后端查询
- 只有在必要时才查询后端

## 五、约束条件

### 5.1 技术约束

- 不能修改后端 API 接口（除非必要）
- 必须保持向后兼容
- 必须支持现有的 URL 类型（Base64、HTTP URL）

### 5.2 业务约束

- 不能影响现有的上传流程
- 不能影响其他模式的附件显示

---

## 六、验收标准

### 6.1 功能验收

1. ✅ **Base64 URL 直接显示**：
   - 传入 Base64 URL 时，不查询后端
   - 图片立即显示在画布上

2. ✅ **HTTP URL 直接显示**：
   - 传入 HTTP URL 时，不查询后端（除非必要）
   - 图片立即显示在画布上

3. ✅ **上传完成后的优化**：
   - 如果上传已完成，可以查询后端获取云 URL
   - 但不应该阻塞初始显示

### 6.2 性能验收

1. ✅ **响应时间**：点击按钮后，图片在 100ms 内显示
2. ✅ **网络请求**：减少不必要的后端查询（至少减少 50%）

### 6.3 可靠性验收

1. ✅ **可用性**：无论上传状态如何，都应该能够显示图片
2. ✅ **容错性**：如果 URL 不可用，显示友好的错误提示

## 七、优先级排序

| 需求 | 优先级 | 原因 |
|------|--------|------|
| R1：优化附件 URL 处理逻辑 | 高 | 核心功能，直接影响用户体验 |

| R3：处理 HTTP 临时 URL 过期问题 | 中 | 提升可靠性 |
| R4：优化查找逻辑 | 中 | 提升可靠性 |
