## 1. 项目背景

### 1.1 现状问题

当前附件处理架构存在以下严重问题：

1. **前后端职责混乱**
   - 前端 `attachmentUtils.ts` 达到 1016 行，承担了大量不应由前端处理的逻辑
   - 后端附件处理逻辑分散在 4 个文件中，缺乏统一服务

2. **重复功能严重**
   - 17 个重复功能点（URL类型判断、HTTP下载、Base64编解码等）
   - Tongyi 模式：前端下载图片 2 次（显示 1 次 + 上传准备 1 次）
   - Google 模式：Base64 数据双向传输（后端→前端 1.33MB + 前端→后端 1.33MB）

3. **性能浪费**
   - 总延迟浪费：~850ms（Tongyi +650ms, Google +200ms）
   - 网络请求浪费：每次操作多 2-4 次请求
   - 数据传输浪费：每张图片多传 2-3.33MB

4. **❌ 关键缺陷：Base64 传输问题**
   - Google 返回 Base64 Data URL（1MB 图片 → 1.33MB，+33%体积）
   - 前端接收 1.33MB Base64
   - 前端再上传 1.33MB 给后端
   - **浪费 2.66MB 传输，且无法用于页面重载**

5. **❌ 关键缺陷：重载缺失图片**
   - 显示URL（临时URL、Blob URL、Base64）无法持久化
   - 没有后端异步上传 → 数据库 url 字段为空
   - 页面重载时无法加载对话历史中的图片

### 1.3 数据流转问题

#### 当前Gen模式数据流（Tongyi）:
```
Google Gemini API 返回 HTTP URL
    ↓
[前端] 下载1: fetch(url) → Blob (显示)            // ❌ 第1次下载
    ↓
[前端] 下载2: sourceToFile → fetch(url) → File  // ❌ 第2次下载
    ↓
[前端] FormData上传: File → 后端
    ↓
[后端] 保存临时文件: File → /tmp/uploads/
    ↓
[Worker Pool] 读取临时文件 → 上传云存储
    ↓
[Worker Pool] 删除临时文件
```

**问题**:
- ❌ **2次网络下载**（前端2次）
- ❌ **1次FormData上传**（前端→后端，可能很大）
- ❌ **1次临时文件IO**（后端保存→读取→删除）
- ❌ **前端处理重**（下载、转File、上传）

#### 当前Gen模式数据流（Google Base64）:
```
Google Gemini API 返回 Base64 Data URL
    ↓
[后端 → 前端] 传输 Base64 (~1.33MB for 1MB图片)  // ❌ Base64体积+33%
    ↓
[前端] 直接显示 Base64 Data URL
    ↓
[前端] Base64 → File 对象                         // ❌ CPU密集
    ↓
[前端] FormData上传: File → 后端                  // ❌ 再次传输1.33MB
    ↓
[后端] 保存临时文件
    ↓
[Worker Pool] 读取 → 上传云存储
```

**问题**:
- ❌ **Base64传输体积大**: 1MB图片变1.33MB（+33%）
- ❌ **双向传输Base64数据**: 后端→前端（1.33MB）+ 前端→后端（1.33MB）
- ❌ **前端CPU浪费**: Base64解码和File对象创建
- ❌ **无法直接用于重载**: Base64不能存数据库，重载时没有图片

#### 当前Edit模式数据流（CONTINUITY LOGIC）:
```
用户在画布看到图片 (activeImageUrl)
    ↓
[前端] findAttachmentByUrl(activeImageUrl, messages)  // 遍历所有消息
    ↓
[前端] tryFetchCloudUrl(sessionId, attachmentId)     // API查询
    ↓
GET /api/sessions/{sessionId}/attachments/{attachmentId}
    ↓
[后端] 查询数据库 → 返回云URL
    ↓
[前端] 使用云URL
```

**问题**:
- ❌ **前端遍历消息**（可能很多）
- ❌ **额外API调用**（查询云URL）
- ❌ **前端逻辑重**（查找、查询、判断）

### 1.4 职责划分混乱

#### 前端当前职责（过重）:
1. ✅ **应该做**: 用户交互、文件选择、显示预览
2. ❌ **不该做**:
   - URL类型判断（17次调用）
   - HTTP URL下载（2次，Tongyi）
   - Base64编解码
   - File对象创建和上传
   - 历史附件查找和云URL查询
   - 附件清理逻辑
   - 3层降级策略

**前端代码量**: `attachmentUtils.ts` **1016行**，过于庞大

#### 后端当前职责（分散）:
1. ✅ **应该做**: 附件统一处理、云存储管理、数据持久化
2. ❌ **当前分散**:
   - 临时文件管理（`storage.py`）
   - Worker Pool处理（`upload_worker_pool.py`）
   - 云URL保护（`sessions.py`）
   - 附件下载（`conversational_image_edit_service.py`）

**后端缺失**: 统一的附件处理服务

### 1.5 模式不一致

| 方面 | Gen模式 | Edit模式 | 不一致点 |
|------|---------|---------|---------|
| **附件来源** | AI生成（无用户上传） | 用户上传 + CONTINUITY LOGIC | 处理逻辑完全不同 |
| **前端处理** | processMediaResult | processUserAttachments | 2个独立函数 |
| **临时URL** | Blob URL或Base64 | Blob URL、Base64、HTTP | 类型更多 |
| **历史查找** | 不需要 | findAttachmentByUrl | Edit独有 |
| **云URL查询** | 不需要 | tryFetchCloudUrl | Edit独有 |

**问题**: 两种模式的附件处理逻辑完全独立，无法复用。

### 1.6 性能问题总结

基于性能分析章节的数据：

| 问题 | 场景 | 浪费 | 影响 |
|------|------|------|------|
| **Tongyi双重下载** | Gen/Edit模式 | 2次网络请求 | 用户等待**+650ms** |
| **Google Base64传输** | Gen/Edit模式 | +33%数据量（1MB→1.33MB） | 传输时间**+200ms** |
| **前端FormData上传** | 所有模式 | 上传大文件 | 网络开销大 |
| **临时文件IO** | 所有模式 | 磁盘读写 | 服务器IO压力 |
| **前端CPU处理** | 所有模式 | Base64编解码、File创建 | 浏览器性能下降 |
| **API额外调用** | Edit CONTINUITY | tryFetchCloudUrl | 增加延迟 |

**总浪费**:
- 网络请求: Tongyi多**2次下载** + **1次API查询**
- 传输数据: Google多**33%** (Base64编码)
- 处理时间: 总延迟**+850ms** (Tongyi +650ms + Google +200ms)

### 1.7 业务影响

- **用户体验差**：图片加载慢（+850ms 延迟）
- **流量成本高**：每张图片多传 2-3.33MB
- **维护成本高**：前端代码过于复杂（1016 行）
- **可靠性差**：重载后图片丢失

## 2. 项目目标

### 2.1 核心目标

**统一附件处理到后端，前端轻量化，消除重复功能，提升性能和可靠性**

### 2.2 具体目标

| 目标类型 | 指标 | 当前值 | 目标值 | 改善 |
|---------|------|--------|--------|------|
| **性能** | 平均延迟 | ~850ms 浪费 | -430ms | **-50%** |
| **性能** | 网络请求 | 4-6 次/操作 | 2-3 次/操作 | **-50%** |
| **性能** | 数据传输 | +2-3.33MB/图片 | 0MB 额外传输 | **-67%到-100%** |
| **代码质量** | 前端代码量 | 1016 行 | 400 行 | **-60%** |
| **可靠性** | 重载图片加载 | 失败（无云URL） | 100% 成功 | **P0 要求** |
| **成本** | 带宽成本 | 基线 | -50% | 显著降低 |

### 2.3 设计原则

1. **功能完全保留** ✅
   - 所有现有功能必须保留
   - 用户体验不能降低
   - 向后兼容旧数据

2. **数据表不迁移** ✅
   - 复用现有数据库表
   - 不改变表结构
   - 仅调整字段使用方式

3. **❗ 避免Base64传输** 🔴 P0（关键要求）
   - **能不用Base64就不用**，体积太大（+33%）
   - 后端可以内部处理Base64（解码、存储）
   - 但**绝不向前端传输Base64 Data URL**
   - 改用临时HTTP URL或代理URL

4. **❗ 区分显示URL和云URL** 🔴 P0（关键要求）
   - **显示URL**: 临时URL，用于前端立即显示（可以是代理URL、临时HTTP URL）
   - **云URL**: 永久云存储URL，用于数据库保存和页面重载
   - 两者必须明确区分，字段分开存储

5. **❗ 重载必须用云URL** 🔴 P0（关键要求）
   - 页面重载时，**必须从云URL加载图片**
   - 显示URL可能已过期，不能用于重载
   - 数据库只保存云URL（url字段）
   - 重载逻辑: `SELECT url FROM message_attachments WHERE upload_status='completed'`

6. **❗ 后端异步上传必须** 🔴 P0（关键要求）
   - **没有后端异步上传，重载就没有图片**
   - Worker Pool异步上传是非协商需求
   - 上传完成前，前端可以显示临时URL
   - 上传完成后，更新数据库url字段为云URL

7. **前端轻量化** 🎯
   - 前端只做UI和用户交互
   - 附件处理逻辑全部后移
   - 减少前端代码量**60%以上**

8. **后端统一化** 🎯
   - 所有模式使用统一的附件处理服务
   - 消除重复代码
   - 职责清晰

9. **性能优化** 🎯
   - 消除重复下载
   - 减少数据传输
   - 降低延迟**70%**

### 3.2 新增功能需求

#### FR-101: 避免 Base64 传输
**优先级**: P0 ❗ **关键要求**
**描述**: 能不用 Base64 就不用，绝不向前端传输 Base64 Data URL

**实现方式**:
- Google 返回 Base64 → 后端创建临时代理 URL `/api/temp-images/{id}`
- 前端请求代理 URL → 后端读取 temp_url → Base64 解码 → 返回图片字节流
- Tongyi 返回 HTTP URL → 直接传递给前端

**验收标准**:
- ✅ Google 模式：前端接收的是 `/api/temp-images/{id}` 而非 Base64
- ✅ 代理端点正常工作，能返回图片字节流
- ✅ 浏览器能正确显示通过代理加载的图片
- ✅ 数据传输量：1MB 图片 < 1.1MB（不能是 1.33MB Base64）

#### FR-102: 区分显示 URL 和云 URL
**优先级**: P0 ❗ **关键要求**
**描述**: 明确区分临时显示 URL 和永久云存储 URL

**数据库字段**:
- `temp_url`: 临时URL（Base64/HTTP临时URL），可清空
- `url`: 云存储URL（永久），不能清空

**验收标准**:
- ✅ 数据库 message_attachments 表有 temp_url 和 url 两个字段
- ✅ 上传前：temp_url 有值，url 为空
- ✅ 上传后：url 有值（云URL），temp_url 可清空
- ✅ API 响应明确标识 URL 类型

#### FR-103: 重载必须用云 URL
**优先级**: P0 ❗ **关键要求**
**描述**: 页面重载时，必须从云存储URL加载图片

**验收标准**:
- ✅ GET /api/sessions/{id}/messages 只返回 upload_status='completed' 的附件
- ✅ 返回的 attachment.url 一定是云存储 URL
- ✅ 前端重载后能显示所有历史图片
- ✅ 测试：关闭浏览器 → 重新打开 → 所有图片正常显示

#### FR-104: 后端异步上传必须
**优先级**: P0 ❗ **关键要求**
**描述**: Worker Pool 异步上传到云存储是非协商需求

**验收标准**:
- ✅ AI 返回图片后立即提交 Worker Pool 任务
- ✅ Worker Pool 成功上传到云存储
- ✅ 上传完成后更新 message_attachments.url 为云URL
- ✅ 上传完成后更新 upload_status='completed'
- ✅ 上传失败有重试机制（最多 3 次）
- ✅ 上传失败有告警（失败率 > 1%）

#### FR-105: 统一附件处理服务
**优先级**: P1
**描述**: 创建统一的 AttachmentService，所有模式复用

**验收标准**:
- ✅ 创建 `attachment_service.py` 文件
- ✅ 提供 4 个主要方法：
  - `process_user_upload()` - 处理用户上传
  - `process_ai_result()` - 处理 AI 返回
  - `resolve_continuity_attachment()` - CONTINUITY LOGIC
  - `get_cloud_url()` - 获取云 URL
- ✅ Gen/Edit 模式都使用统一服务
- ✅ 无重复代码

## 4. 非功能需求

**验收标准**:

**具体指标**:

**验收标准**:
- ✅ 网络监控显示请求次数符合目标

#### NFR-003: 数据传输减少

**验收标准**:
- ✅ 抓包分析显示数据量符合目标

### 4.2 可靠性需求

- ✅ 监控显示上传成功率 > 99%

## 5. 约束和假设

### 5.1 技术约束

1. **数据库不迁移**
   - 必须使用现有数据库表结构
   - 只能增加字段，不能删除或修改现有字段
   - 现有数据必须向后兼容

2. **API向后兼容**
   - 旧的 FormData 上传 API 必须保留
   - 旧的 Worker Pool 任务类型必须保留
   - 前端可以逐步迁移，但后端必须同时支持新旧两种方式

3. **浏览器兼容性**
   - 支持现代浏览器（Chrome, Firefox, Safari, Edge）
   - 不支持 IE11

### 5.2 业务约束

1. **功能完全保留**
   - 所有现有功能必须保留
   - 用户体验不能降低
   - 不能影响现有业务流程

2. **性能不能降低**
   - 新架构的性能必须优于或等于当前架构
   - 关键指标（延迟、成功率）不能下降

### 5.3 假设

1. **Worker Pool 正常运行**
   - 假设 Worker Pool 服务稳定，队列不会堵塞
   - 假设云存储服务可用性 > 99.9%

2. **网络环境稳定**
   - 假设前后端网络连接稳定
   - 假设云存储网络连接稳定

3. **数据量不会过大**
   - 假设单次操作的图片数量 < 10 张
   - 假设单张图片大小 < 10MB

## 7. 风险和假设

### 7.1 潜在风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| **后端API延迟增加** | 中 | 中 | Feature flag快速回滚 |
| **Worker Pool负载增加** | 低 | 中 | 增加Worker数量，优化队列 |

| **前端显示问题** | 中 | 低 | A/B测试，逐步迁移 |
| **数据迁移失败** | 低 | 高 | 保留旧数据，分批迁移 |

### 7.2 监控指标

**必须监控**:
1. API响应时间（P50, P90, P99）
2. Worker Pool队列长度
3. 上传成功率
4. 前端错误率
5. 用户投诉数量

**告警阈值**:
- API P99延迟 > 2000ms
- Worker Pool队列 > 100
- 上传失败率 > 1%
- 前端错误率 > 0.5%

   - 监控关键指标

2. **数据修复**
   - 检查数据库数据完整性
