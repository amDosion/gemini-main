# 附件查找逻辑说明

## 概述

附件查找（CONTINUITY LOGIC）应该由**后端统一处理**，前端只作为降级方案。

## 架构设计

### 后端统一处理（推荐）

**API端点**：`POST /api/attachments/resolve-continuity`

**职责**：
- ✅ 在数据库中查找匹配的附件
- ✅ 检查附件上传状态（`upload_status`）
- ✅ 返回云URL（如果已上传完成）
- ✅ 创建上传任务（如果未上传）

**优势**：
- 数据权威性：后端直接查询数据库，获取最新状态
- 避免重复上传：检查 `upload_status == 'completed'` 且有 HTTP URL 时，直接返回，不创建新任务
- 统一处理：所有附件查找逻辑集中在一个地方

### 前端降级方案（向后兼容）

**函数**：`findAttachmentByUrl()`

**用途**：
- 当后端API调用失败时的降级方案
- 向后兼容旧代码

**限制**：
- 只能查找前端 `messages` 数组中的附件
- 无法获取数据库中的最新状态
- 可能返回过时的信息

## 当前实现

### 前端流程

在 `prepareAttachmentForApi()` 中：

1. **优先使用后端API**：
   ```typescript
   // 步骤 1: 优先使用后端CONTINUITY API
   const response = await fetch('/api/attachments/resolve-continuity', {
     method: 'POST',
     body: JSON.stringify({
       activeImageUrl: imageUrl,
       sessionId: sessionId,
       messages: messages  // 可选
     })
   });
   ```

2. **降级到前端查找**：
   ```typescript
   // 步骤 2: 降级方案 - 前端查找（向后兼容）
   const found = findAttachmentByUrl(imageUrl, messages);
   ```

### 后端流程

在 `AttachmentService.resolve_continuity_attachment()` 中：

1. **查找附件ID**：在 `messages` 中查找匹配的附件ID
2. **查询数据库**：根据附件ID查询数据库获取最新状态
3. **检查上传状态**：
   - 如果 `upload_status == 'completed'` 且有 HTTP URL → 直接返回，不创建新任务
   - 如果未上传 → 创建上传任务

## 最佳实践

### ✅ 推荐做法

1. **优先使用后端API**：
   - 调用 `/api/attachments/resolve-continuity`
   - 后端返回最新的附件状态和云URL

2. **检查返回状态**：
   - 如果 `status === 'completed'` 且有 `url` → 直接使用云URL
   - 如果 `status === 'pending'` → 等待上传完成

### ⚠️ 注意事项

1. **前端查找的局限性**：
   - 只能查找前端 `messages` 数组中的附件
   - 无法获取数据库中的最新状态
   - 可能返回过时的信息

2. **数据一致性**：
   - 后端查询数据库，获取最新状态
   - 前端 `messages` 可能不是最新的

3. **避免重复上传**：
   - 后端会检查 `upload_status == 'completed'` 且有 HTTP URL
   - 如果已上传完成，直接返回，不创建新任务

## 总结

**附件查找应该由后端进行**，因为：
1. ✅ 后端可以直接查询数据库，获取最新状态
2. ✅ 后端可以检查上传状态，避免重复上传
3. ✅ 后端可以统一处理所有附件查找逻辑

前端 `findAttachmentByUrl()` 只作为降级方案，用于向后兼容。
