# 图像编辑服务架构重构总结

## 一、重构背景

### 1.1 原始问题

项目中图像编辑服务存在架构不统一的问题：

- `image_edit_vertex_ai.py` 是一个"大一统"文件，`VertexAIImageEditor` 类同时处理 inpainting、background-edit、recontext 三种模式，没有按模式拆分
- `MaskEditService` 使用**无状态接口**（空构造函数 `__init__(self)`，每次调用时传递凭据），与其他服务的有状态模式不一致
- `ImageEditCoordinator` 中 Route 2（mask-edit）需要 ~50 行手动参数提取和凭据传递代码，而 Routes 3-5 只需 3 行
- `MaskEditService` 完全覆写 `edit_image()` 方法，不复用基类的 `_build_config()` / `_build_reference_images()` / `_process_response()` 共享管线

### 1.2 参考架构（gen 模式）

`VertexAIImageGenerator`（gen 模式）代表了正确的架构模式：

```
构造函数绑定凭据 → 协调器创建并缓存实例 → 统一接口调用
```

```python
# imagen_vertex_ai.py
class VertexAIImageGenerator:
    def __init__(self, project_id, location, credentials_json):  # 凭据在构造时绑定
        self._client = None
        self._initialized = False

    def _ensure_initialized(self):  # 懒加载客户端

    async def generate_image(self, prompt, model, **kwargs):  # 统一接口
```

协调器中的使用方式（3 行代码）：
```python
generator = self.get_generator()  # 从缓存获取或创建
return await generator.generate_image(prompt, model, **kwargs)
```

### 1.3 重构目标

1. 将 `image_edit_vertex_ai.py` 拆分为独立的服务文件，每个模式一个文件
2. 统一所有 Vertex AI 编辑服务的接口，与 gen 模式保持一致
3. 消除 Coordinator 中 mask-edit 路由的特殊处理代码
4. **所有服务（包括 MaskEditService）必须复用基类的共享管线**

### 2.2 重构文件

| `vertexai/mask_edit_service.py` | 从完全覆写 `edit_image()` 改为**薄包装器**：设置掩码编辑默认值 → `super().edit_image()`；删除所有内部方法（`_edit_with_mask` / `_edit_without_mask` / `_handle_error` 等） |

### 2.4 MaskEditService 薄包装器

```python
class MaskEditService(VertexAIEditBase):
    DEFAULT_EDIT_MODE = 'mask_edit'  # → EDIT_MODE_INPAINT_INSERTION

    def edit_image(self, prompt, reference_images, config=None):
        # 1. 验证 raw 图片存在
        # 2. 设置掩码编辑默认值：
        #    guidance_scale=15.0, output_mime_type='image/png',
        #    output_compression_quality=95,
        #    safety_filter_level='BLOCK_MEDIUM_AND_ABOVE',
        #    person_generation='ALLOW_ADULT'
        # 3. 无用户掩码时自动设置 mask_mode='MASK_MODE_FOREGROUND'
        # 4. 委托 → super().edit_image(prompt, reference_images, effective_config)
        #    基类执行完整共享管线
```

所有内部方法（`_edit_with_mask`、`_edit_without_mask`、`_clean_base64`、`_bytes_to_base64`、`_base64_to_bytes`、`_handle_error`）已删除，`EditResult` 数据类保留用于向后兼容。

### 2.5 Coordinator 统一路由模式

重构前（路由 2 vs 路由 3-5 差异巨大）：

```python
# 路由 2: mask-edit — 50 行特殊处理
mask_service = MaskEditService()  # 空构造函数
project_id = self._config.get('vertex_ai_project_id')
location = self._config.get('vertex_ai_location', 'us-central1')
credentials_json = self._config.get('vertex_ai_credentials_json')
# ... 手动提取 raw、mask、edit_mode、mask_dilation 等
result = mask_service.edit_with_mask(
    image_base64=image_base64, mask_base64=mask_base64, prompt=prompt,
    project_id=project_id, location=location, credentials_json=credentials_json, ...
)
# ... EditResult → List[Dict] 转换

# 路由 3-5: inpainting/background-edit/recontext — 3 行 + editMode 归一化
if 'editMode' in kwargs and kwargs['editMode']:
    kwargs['edit_mode'] = kwargs.pop('editMode')
editor = self.get_editor()
return editor.edit_image(prompt=prompt, reference_images=reference_images, config=kwargs)
```

重构后（路由 2-5 **完全一致**的 3 行模式）：

```python
# 路由 2: mask-edit
self._require_vertex_ai(mode)
kwargs['model'] = self._validate_edit_model(model)
editor = self.get_mask_editor()                    # → _get_cached_service('mask_edit', MaskEditService)
return editor.edit_image(prompt, reference_images, config=kwargs)

# 路由 3: inpainting
self._require_vertex_ai(mode)
kwargs['model'] = self._validate_edit_model(model)
editor = self.get_inpainting_editor()              # → _get_cached_service('inpainting', InpaintingService)
return editor.edit_image(prompt, reference_images, config=kwargs)

# 路由 4: background-edit
self._require_vertex_ai(mode)
kwargs['model'] = self._validate_edit_model(model)
editor = self.get_background_editor()              # → _get_cached_service('background_edit', BackgroundEditService)
return editor.edit_image(prompt, reference_images, config=kwargs)

# 路由 5: recontext
self._require_vertex_ai(mode)
kwargs['model'] = self._validate_edit_model(model)
editor = self.get_recontext_editor()               # → _get_cached_service('recontext', RecontextService)
return editor.edit_image(prompt, reference_images, config=kwargs)
```

路由 3-5 不再有 `editMode → edit_mode` 归一化步骤，因为基类 `_build_config()` 已原生支持 `editMode` 和 `edit_mode` 双格式。

Coordinator 辅助方法：

```python
def _get_cached_service(self, cache_key, service_class):
    """统一的服务创建/缓存（凭据从 _config 提取）"""
    if cache_key in self._editor_cache:
        return self._editor_cache[cache_key]
    project_id, location, credentials_json = self._get_vertex_credentials()
    service = service_class(project_id=project_id, location=location, credentials_json=credentials_json)
    self._editor_cache[cache_key] = service
    return service
```

### 3.3 MaskEditService 的差异化（仅为默认值）

MaskEditService 现在是纯粹的薄包装器，唯一的差异是**设置默认参数**：

| 参数 | MaskEditService 默认值 | 基类默认值 | 差异原因 |
|------|----------------------|----------|---------|
| `guidance_scale` | `15.0` | 无 | 掩码编辑需要较高的引导比例 |
| `output_mime_type` | `image/png` | `image/jpeg` | 掩码编辑通常需要无损格式 |
| `output_compression_quality` | `95` | 无 | 掩码编辑需要高质量输出 |
| `safety_filter_level` | `BLOCK_MEDIUM_AND_ABOVE` | 无 | 掩码编辑的安全默认值 |
| `person_generation` | `ALLOW_ADULT` | 无 | 掩码编辑允许成人人物 |
| 无掩码时 `mask_mode` | `MASK_MODE_FOREGROUND` | 无 | 自动前景掩码 |

这些默认值通过 `setdefault()` 设置，用户传入的值会覆盖默认值。

## 五、与 gen 模式的架构对比

```
gen 模式完整流程：
  Frontend → modes.py → GoogleService.generate_image()
  → ImagenCoordinator._load_config() (DB → 解密凭据 → env 回退)
  → ImagenCoordinator.get_generator() (缓存)
  → VertexAIImageGenerator(project_id, location, credentials_json)
  → generator.generate_image(prompt, model, **kwargs)
  → List[Dict[str, Any]]

edit 模式完整流程（路由 2-5，重构后）：
  Frontend → modes.py → GoogleService.edit_image()
  → ImageEditCoordinator._load_config() (DB → 解密凭据 → env 回退)
  → ImageEditCoordinator.get_xxx_editor() (缓存)
  → XxxService(project_id, location, credentials_json)
  → editor.edit_image(prompt, reference_images, config)
  →   _build_config() → _build_reference_images() → API 调用 → _process_response()
  → List[Dict[str, Any]]
```

两条流程在**所有层级**已完全统一：
- **配置加载**：`_load_config()` 模式相同（DB 优先 → 解密 → env 回退）
- **服务创建**：`_get_cached_service()` 模式相同（缓存 → 凭据三元组 → 构造函数）
- **接口签名**：构造函数 `(project_id, location, credentials_json)`，业务方法返回 `List[Dict[str, Any]]`
- **内部管线**：所有服务共用 `_build_config()` + `_build_reference_images()` + `_process_response()`
- **错误处理**：所有服务共用基类的分类错误处理逻辑
- **唯一合理差异**：gen 模式调用 `generate_image()`，edit 模式调用 `edit_image()`（业务方法名不同是合理的）
