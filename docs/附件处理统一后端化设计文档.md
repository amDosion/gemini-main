# 附件处理统一后端化设计文档

## 1. 问题分析

### 1.1 当前问题

经过多轮分析，发现附件处理存在大量重复功能：

#### 前端重复处理
1. **`processUserAttachments`** (`attachmentUtils.ts:786-942`)
   - Base64 URL → File 对象转换
   - Blob URL → Base64/File 对象转换
   - HTTP URL 处理
   - 云存储 URL 查询 (`tryFetchCloudUrl`)

2. **`prepareAttachmentForApi`** (`attachmentUtils.ts:653-758`)
   - 历史附件查找 (`findAttachmentByUrl`)
   - 云存储 URL 查询 (`tryFetchCloudUrl`)
   - URL 类型判断和转换

3. **`processMediaResult`** (`attachmentUtils.ts:960-1016`)
   - HTTP URL 下载和 Blob URL 创建
   - 附件上传任务创建

#### 后端重复处理
1. **`convert_attachments_to_reference_images`** (`modes.py:102-145`)
   - 附件字段提取（url > tempUrl > fileUri > base64Data）
   - 转换为 reference_images 字典

2. **`_convert_reference_images`** (`conversational_image_edit_service.py:711-783`)
   - URL 类型判断（HTTP/Base64/纯 Base64）
   - 转换为 SDK 需要的格式

#### 重复逻辑
- ✅ URL 类型判断（Base64/Blob/HTTP）
- ✅ URL 转换（Blob → Base64, Base64 → File, HTTP → download）
- ✅ 附件查找和复用
- ✅ 云存储 URL 查询
- ✅ 附件状态管理

### 1.2 问题影响

1. **代码复杂度高**：前端有大量附件处理逻辑（1000+ 行）
2. **维护困难**：前后端都有类似逻辑，修改需要同步
3. **性能问题**：前端需要下载临时 URL 并转换，影响用户体验
4. **不一致风险**：前后端处理逻辑可能不一致，导致 bug
5. **测试困难**：需要同时测试前后端逻辑

## 2. 设计目标

### 2.1 核心目标

1. **统一处理**：所有附件处理逻辑统一在后端（**仅针对 API 调用时的转换**）
2. **简化前端**：前端只负责收集附件信息，不做复杂转换（**用于 API 调用**）
3. **性能优化**：减少前端下载和转换操作（**临时 URL 可直接使用**）
4. **易于维护**：单一数据源，逻辑集中
5. **⚠️ 保持显示逻辑不变**：前端显示仍然使用 `att.url` 字段，区分两种情况：
   - 未重载时：Base64/临时 URL（从内存/状态）
   - 重载后：永久云 URL（从数据库加载）

### 2.2 设计原则

1. **最小化前端改动**：保持现有 API 接口，只改变处理方式
2. **向后兼容**：支持现有附件格式，逐步迁移
3. **性能优先**：减少不必要的下载和转换
4. **错误处理**：统一的错误处理和日志
5. **⚠️ 保持显示逻辑不变**：
   - 前端显示仍然使用 `att.url` 字段
   - 区分"未重载"（Base64/临时 URL）和"重载后"（永久云 URL）
   - 确保重载网页后功能完全正常

## 4. 实现方案

### 4.1 前端改动 ⚠️ 重要：显示逻辑保持不变

#### 4.1.0 显示逻辑说明（关键）

**显示逻辑需要区分两种情况**：

1. **未重载网页时（首次生成）**：
   - 显示 AI 生成的图片对应的 **Base64 或临时 URL**
   - `att.url` 包含：Base64 Data URL 或 HTTP 临时 URL
   - 从内存/状态中获取，立即显示

2. **重载网页后**：
   - 显示永久云存储 URL
   - `att.url` 包含：永久云存储 HTTP URL
   - 从数据库加载，通过 `MessageAttachment.to_dict()` 获取

**当前实现（功能正常）**：
```typescript

        self,
        attachments: List[Dict[str, Any]],

        user_id: str,

        # 返回临时 URL 或云存储 URL

    async def _normalize_url(
        self,
        attachment: Dict[str, Any],
        session_id: str

        """标准化 URL"""
        url = attachment['url']
        
        if url.startswith('data:'):
            # Base64 Data URL
            return await self._normalize_base64_url(url, attachment)

        return NormalizedAttachment(
            id=attachment.get('id', ''),
            base64_data=base64_data,
            mime_type=attachment.get('mimeType', 'image/png'),
            name=attachment.get('name', ''),
            role=attachment.get('role'),
            upload_status='pending'
        )
    
    async def lookup_and_reuse_attachments(
        self,
        attachments: List[NormalizedAttachment],
        session_id: str,
        user_id: str
    ) -> List[NormalizedAttachment]:
        """附件查找和复用"""
        reused = []
        
        for att in attachments:
            # 通过 temp_url 查找历史附件
            if att.temp_url:
                found = await self.lookup_service.find_by_url(
                    att.temp_url, session_id, user_id
                )
                if found:
                    # 获取云存储 URL
                    cloud_url = await self.lookup_service.get_cloud_url(
                        found.id, session_id
                    )
                    if cloud_url:
                        # 复用云存储 URL
                        att.url = cloud_url
                        att.upload_status = 'completed'
            
            reused.append(att)
        
        return reused
    
    async def convert_to_reference_images(
        self,
        attachments: List[NormalizedAttachment],
        provider: str
    ) -> Dict[str, Any]:
        """转换为 reference_images 格式"""
        reference_images = {}
        
        for att in attachments:
            # 根据提供商类型选择数据源
            if provider == 'google':
                # Google: 优先使用 file_uri，然后 base64_data，最后 url
                if att.file_uri:
                    image_data = att.file_uri
                elif att.base64_data:
                    image_data = f"data:image/png;base64,{att.base64_data}"
                elif att.url:
                    image_data = att.url
                else:
                    continue
            else:
                # 其他提供商：优先使用 url，然后 base64_data
                if att.url:
                    image_data = att.url
                elif att.base64_data:
                    image_data = f"data:image/png;base64,{att.base64_data}"
                else:
                    continue
            
            # 根据 role 设置键名
            key = att.role if att.role else 'raw'
            if key == 'mask':
                reference_images['mask'] = image_data
            else:
                if 'raw' not in reference_images:
                    reference_images['raw'] = image_data
                else:
                    # 多个 raw 图片，使用列表
                    if isinstance(reference_images['raw'], list):
                        reference_images['raw'].append(image_data)
                    else:
                        reference_images['raw'] = [reference_images['raw'], image_data]
        
        return reference_images
```

### 5.5.5 关键设计点总结

**✅ 设计原则**：

1. **前端显示逻辑保持不变**：
   - 始终使用 `att.url` 字段
   - 支持 Base64/临时 URL（未重载）和永久云 URL（重载后）
   - 浏览器自动处理各种 URL 类型的显示

2. **后端统一处理主要针对 API 调用**：
   - 前端发送附件到后端 API 时，后端统一处理转换
   - 不影响前端的显示逻辑

3. **数据库保存时区分**：
   - `url` 字段：永久云 URL（用于重载后显示）
   - `temp_url` 字段：临时 URL（用于跨模式查找）

4. **重载后功能完全正常**：
   - 从数据库加载的 `att.url` 是永久云 URL
   - 直接使用 `att.url` 显示，功能完全正常 ✅
