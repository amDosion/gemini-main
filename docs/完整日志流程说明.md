# 完整日志流程说明

## 概述

本文档说明从前端请求到后端响应的完整日志链，确保每个步骤都有清晰的开始和结束日志，形成环环相扣的日志流程。

## 日志输出方式

所有关键日志都使用**双重输出**：
1. **logger.info()** - 通过 Python logging 模块输出（会传播到根 logger）
2. **print(..., file=sys.stderr, flush=True)** - 直接输出到 stderr（确保终端能看到）

这样可以确保即使 logging 配置有问题，日志也能在终端显示。

## 完整日志链

### 1. [Request] 中间件层
- **位置**: `backend/app/main.py` - `RequestLoggingMiddleware`
- **日志**:
  - `[Request] ========== 收到请求 ==========`
  - `[Request] 📥 {method} {path}`
  - `[Request] ✅ 请求处理完成 (耗时: Xms, 状态码: XXX)`

### 2. [Auth] 认证层
- **位置**: `backend/app/core/user_context.py` - `require_user_id()`
- **日志**:
  - `[Auth] 🔐 开始认证检查: {method} {path}`
  - `[Auth] ✅ 认证成功: user_id={user_id}...` 或 `[Auth] ❌ 认证失败`

### 3. [Modes] 路由处理层
- **位置**: `backend/app/routers/core/modes.py` - `handle_mode()`
- **日志步骤**:
  - **开始**: `[Modes] ========== 开始处理模式请求 ==========`
  - **步骤1**: `[Modes] 🔄 [步骤1] 获取提供商凭证...` → `[Modes] ✅ [步骤1] 凭证获取完成`
  - **步骤2**: `[Modes] 🔄 [步骤2] 创建提供商服务...` → `[Modes] ✅ [步骤2] 服务创建完成`
  - **步骤3**: `[Modes] 🔄 [步骤3] 获取服务方法名...` → `[Modes] ✅ [步骤3] 方法名: {method_name}`
  - **步骤4**: `[Modes] 🔄 [步骤4] 检查服务是否支持方法...` → `[Modes] ✅ [步骤4] 服务支持方法`
  - **步骤5**: `[Modes] 🔄 [步骤5] 准备调用参数...` → `[Modes] ✅ [步骤5] 参数准备完成`
  - **步骤6**: `[Modes] 🔄 [步骤6] 调用服务方法...` → `[Modes] ✅ [步骤6] 服务方法调用完成`
  - **步骤7**: `[Modes] 🔄 [步骤7] 处理图片生成/编辑结果...` → `[Modes] ✅ [步骤7] 所有图片处理完成`
  - **结束**: `[Modes] ========== 模式请求处理完成 (总耗时: Xms) ==========`

### 4. [CredentialManager] 凭证获取层
- **位置**: `backend/app/core/credential_manager.py` - `get_provider_credentials()`
- **日志**:
  - `[CredentialManager] 🔄 开始获取凭证: provider={provider}, user_id={user_id}...`
  - `[CredentialManager] ✅ 使用激活配置 '{profile_name}' (耗时: Xms)` 或
  - `[CredentialManager] ✅ 使用配置 '{profile_name}' (回退, 耗时: Xms)`

### 5. [ProviderFactory] 服务创建层

  - `[ProviderFactory] ✅ 服务创建完成 (耗时: Xms), 类型: {service_type}`

### 6. [GoogleService] 服务协调层

- **日志**:

## 日志格式

所有日志都遵循以下格式：
```
[模块名] 🔄 [步骤X] 操作描述...
[模块名] ✅ [步骤X] 操作完成 (耗时: X.XXms)
[模块名] ❌ [步骤X] 操作失败: 错误信息
```

## 关键特性

1. **双重输出**: 每个关键日志都同时使用 `logger.info()` 和 `print(..., file=sys.stderr, flush=True)`
2. **步骤编号**: 每个步骤都有明确的编号（步骤1、步骤2等）
3. **耗时统计**: 每个步骤都记录耗时，便于性能分析
4. **参数详情**: 关键参数都会记录（model、prompt、user_id等）
5. **错误处理**: 所有异常都有详细的错误日志

## 测试方法

使用测试脚本验证日志输出：
```bash
cd backend
python test_request_flow.py
```

脚本会：
1. 自动登录获取 token
2. 发送图片生成请求
3. 观察后端终端输出

## 注意事项

1. **终端编码**: Windows 控制台可能需要设置 UTF-8 编码
2. **日志级别**: 确保所有 logger 的级别设置为 INFO 或更低
3. **日志传播**: 确保所有子 logger 的 `propagate=True`
4. **输出流**: 使用 `sys.stderr` 而不是 `sys.stdout`（Uvicorn 默认使用 stderr）
