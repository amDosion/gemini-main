# 开发服务器管理规范

## 概述

本文档定义 Kiro 如何管理项目的开发服务器（Dev Servers），包括启动、监视、停止等操作规范。

**核心原则**：Kiro 自动管理后台进程，让开发者专注于编码，无需手动切换终端窗口。

---

## 1. 项目启动命令

### 1.1 后端服务

**启动命令**：
```bash
pip install -r backend/requirements.txt && uvicorn backend.app.main:app --host 0.0.0.0 --port 21574 --reload
```

**工作目录**：`D:\gemini-main\gemini-main`

**端口**：21574

**特性**：
- 自动重载（--reload）
- 监听所有网络接口（0.0.0.0）
- 依赖自动安装

### 1.2 前端服务

**启动命令**：
```bash
npx vite
```

**工作目录**：`D:\gemini-main\gemini-main`

**端口**：21573

**特性**：
- 热模块替换（HMR）
- 快速冷启动
- 自动打开浏览器

## 2. Dev Server 工作流程

### 2.1 启动服务器

**用户请求示例**：
- "启动后端服务器"
- "Start the development server"
- "运行前端开发服务"

**Kiro 执行流程**：

```
1. 识别启动请求
   └─ 判断是后端还是前端

2. 检查进程是否已存在
   ├─ 已存在 → 复用现有进程
   └─ 不存在 → 创建新进程

3. 启动后台进程
   └─ controlPwshProcess(
        action="start",
        command="<启动命令>",
        path="D:\\gemini-main\\gemini-main"
      )

4. 返回进程信息
   └─ processId: xxx
   └─ 终端名称: "Kiro: <服务名>"

5. 向用户报告
   └─ "✅ <服务名> 已启动（进程 ID: xxx）"
```

### 2.2 监视进程输出

**用户请求示例**：
- "检查后端服务器输出"
- "前端有没有编译错误？"
- "查看开发服务器状态"

**Kiro 执行流程**：

```
1. 获取进程输出
   └─ getProcessOutput(
        processId=xxx,
        lines=100  # 最近 100 行
      )

2. 分析输出内容
   ├─ 检查启动成功标志
   ├─ 识别编译错误
   ├─ 识别警告信息
   └─ 检查端口冲突

3. 向用户报告
   ├─ 成功：✅ 服务运行正常
   ├─ 错误：❌ 发现编译错误，建议修复
   └─ 警告：⚠️ 存在警告，建议检查
```

### 2.3 列出活动进程

**用户请求示例**：
- "列出所有后台进程"
- "有哪些服务在运行？"
- "显示活动的开发服务器"

**Kiro 执行流程**：

```
1. 获取进程列表
   └─ listProcesses()

2. 过滤开发服务器进程
   └─ 后端：包含 "uvicorn"
   └─ 前端：包含 "vite"

3. 格式化输出
   └─ 进程 ID | 命令 | 工作目录 | 状态
```

### 2.4 停止服务器

**用户请求示例**：
- "停止后端服务器"
- "关闭所有开发服务"
- "终止前端进程"

**Kiro 执行流程**：

```
1. 识别要停止的进程
   └─ 根据服务名或进程 ID

2. 停止进程
   └─ controlPwshProcess(
        action="stop",
        processId=xxx
      )

3. 清理资源
   └─ 关闭终端
   └─ 释放端口

4. 向用户报告
   └─ "✅ <服务名> 已停止"
```

## 3. 进程复用机制

### 3.1 复用规则

**触发条件**：
- 相同的命令
- 相同的工作目录
- 进程仍在运行

**Kiro 行为**：
```
1. 检测到重复启动请求

2. 查找现有进程
   └─ listProcesses()
   └─ 匹配命令和路径

3. 复用现有进程
   └─ 返回现有 processId
   └─ 不创建新进程

4. 向用户报告
   └─ "ℹ️ <服务名> 已在运行（进程 ID: xxx），复用现有进程"
```

### 3.2 避免冲突

**端口冲突检测**：
```
1. 启动前检查端口占用
   └─ 后端：21574
   └─ 前端：21573

2. 发现冲突
   ├─ 是本项目进程 → 复用
   └─ 是其他进程 → 提示用户

3. 建议操作
   └─ "端口 21574 已被占用，是否停止现有进程？"
```

## 4. 自动验证工作流

### 4.1 代码更改后验证

**Steering 规则**：
```markdown
# 开发工作流

代码更改后自动执行：
1. 检查 dev server 输出
2. 查找编译错误或警告
3. 如有错误，建议修复方案
4. 验证通过后继续
```

**Kiro 执行**：
```
1. 用户修改代码

2. Kiro 自动检查
   └─ getProcessOutput(processId=xxx)
   └─ 分析最新输出

3. 发现问题
   ├─ 编译错误 → 提供修复建议
   ├─ 类型错误 → 运行 getDiagnostics
   └─ 警告信息 → 提醒用户注意

4. 验证通过
   └─ "✅ 编译成功，可以继续开发"
```

### 4.2 双重验证

**结合代码诊断**：
```
用户："检查代码是否有问题"

Kiro 执行：
1. 检查 dev server 输出
   └─ 运行时编译问题

2. 运行代码诊断
   └─ getDiagnostics(paths=[...])
   └─ 静态分析问题

3. 综合报告
   ├─ Dev Server: ✅ 编译成功
   ├─ TypeScript: ⚠️ 2 个类型警告
   └─ ESLint: ❌ 1 个 lint 错误

4. 提供修复建议
```

## 6. 故障排除

### 6.1 进程无法启动

**症状**：
- 启动命令失败
- 端口被占用
- 依赖未安装

**解决方案**：
```
1. 检查终端输出
   └─ getProcessOutput(processId=xxx)

2. 验证命令正确性
   └─ 检查路径和参数

3. 确认依赖已安装
   └─ 后端：pip list
   └─ 前端：npm list

4. 检查端口占用
   └─ netstat -ano | findstr :21574
   └─ netstat -ano | findstr :21573
```

### 6.2 无法读取输出

**症状**：
- getProcessOutput 返回空
- 进程状态未知

**解决方案**：
```
1. 确认进程仍在运行
   └─ listProcesses()

2. 检查进程状态
   └─ status: running/stopped/unknown

3. 重启进程
   └─ 停止 → 重新启动
```

### 6.3 多进程冲突

**症状**：
- 端口冲突错误
- 多个相同服务运行

**解决方案**：
```
1. 列出所有进程
   └─ listProcesses()

2. 识别重复进程
   └─ 相同命令和路径

3. 停止不必要的进程
   └─ controlPwshProcess(action="stop")

4. 启用进程复用
   └─ Kiro 自动复用现有进程
```
